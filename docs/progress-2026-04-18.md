## Task 1 — `is_my_game` 플래그

**문제의식**: 기존 구조는 Steam top N 게임 간 상호 유사도만 계산.
PM이 "어떤 게임의 경쟁작"을 분석하라고 지정할 방법이 없었음 → 내부 툴로 부적합.

**변경**
| 파일 | 내용 |
|---|---|
| `schema.sql` | `games.is_my_game BOOLEAN NOT NULL DEFAULT FALSE` + partial index `idx_games_my` |
| `src/gca/cli.py` | `gca add-my-game --platform steam --appid <id>` 서브커맨드 |
| `src/gca/engine/similarity.py` | `compute_all` base 루프를 `is_my_game = TRUE` 게임으로 제한 |
| `src/gca/report/weekly.py` | `generate_all` JOIN 필터 `g.is_my_game = TRUE` |
| `src/gca/api/routes/competitors.py` | `GET /games?mine=true` 쿼리 파라미터 + 응답에 `is_my_game` 포함 |

**DB 마이그레이션 (기존 DB에 적용 완료)**
```sql
ALTER TABLE games ADD COLUMN IF NOT EXISTS is_my_game BOOLEAN NOT NULL DEFAULT FALSE;
CREATE INDEX IF NOT EXISTS idx_games_my ON games (is_my_game) WHERE is_my_game = TRUE;
```

**E2E 검증**
- `gca add-my-game --platform steam --appid 1063730` → `New World: Aeternum` 등록 ✓
- `gca similarity --week-of 2026-04-13` → 이전 90개 → **9개 row만** 기록 (base 필터 정상) ✓
- `curl /games?mine=true` → 단일 게임만 반환 ✓

## Task 3 — FastAPI Vercel 배포 준비

**제약**: 루트 `pyproject.toml`은 `sentence-transformers`, `torch` 포함 → Vercel 함수 50MB 초과.
하지만 API 라우트(`competitors`, `feedback`, `reports`)는 ML deps 미사용 (DB read-only + Jinja2 렌더).
ML은 batch 파이프라인(`extract-features`, `embed`)에만 필요 → 분리 가능.

**생성 파일**
| 경로 | 역할 |
|---|---|
| `api/index.py` | Vercel Python runtime ASGI 엔트리 (`from gca.api.server import app`). `sys.path`에 `src/` 주입 |
| `api/requirements.txt` | ML-free lean deps (fastapi, uvicorn, psycopg[binary], pgvector, pydantic, openai, jinja2, numpy) |
| `vercel.json` (루트) | `@vercel/python` 빌더, `includeFiles: src/**`, 모든 라우트 → `api/index.py` |
| `web/vercel.json` | Next.js framework preset |

**미지원**: `POST /reports/generate`는 similarity → embedder 체인을 트리거 → sentence-transformers 필요.
Vercel에서는 동작하지 않음. PoC 범위에선 로컬/cron batch가 DB에 리포트를 기록하면 됨.

## Task 4 — Terraform IaC

**구조**
```
infra/terraform/
├── main.tf                         # providers + 모듈 호출
├── variables.tf                    # 공통 변수 + sensitive 처리
├── outputs.tf                      # project_ref, database_url, web_url, api_url
├── terraform.tfvars.example        # 값 주입 템플릿
└── modules/
    ├── supabase/                   # supabase_project + pgvector + schema apply
    └── vercel/                     # vercel_project (web + api) + env vars
```

**Providers**
- `supabase/supabase ~> 1.5`
- `vercel/vercel ~> 2.0`
- `cyrilgdn/postgresql ~> 1.22` (alias — 후속 DB 작업용)
- `hashicorp/null ~> 3.2` (스키마 apply provisioner)

**Supabase 모듈 동작**
1. `supabase_project` 생성 (region, org, password 주입)
2. `null_resource.apply_schema` — DB 준비 대기 후 `psql`로 `CREATE EXTENSION vector` + `schema.sql` 적용
3. `filesha256(schema.sql)`을 trigger에 넣어 스키마 변경 시 재실행

**Vercel 모듈 동작**
- `gca-api` 프로젝트 (framework=`other`, root=`/`) + `DATABASE_URL`, `GROQ_API_KEY`, `LLM_MODEL` env
- `gca-web` 프로젝트 (framework=`nextjs`, root=`web`) + `NEXT_PUBLIC_API_BASE_URL` (api 프로젝트 *.vercel.app hostname 참조)
- 둘 다 동일 GitHub repo에서 git 연동

**.gitignore 업데이트**
```
infra/**/.terraform/
infra/**/terraform.tfvars
infra/**/*.tfstate*
infra/**/*.tfplan
web/.next/
web/.vercel/
```

**수동 단계** (apply 전)
```bash
cp infra/terraform/terraform.tfvars.example infra/terraform/terraform.tfvars
# supabase_access_token, supabase_org_id, vercel_api_token, github_repo, groq_api_key 채우기
cd infra/terraform && terraform init && terraform plan
# 확인 후 terraform apply
```
