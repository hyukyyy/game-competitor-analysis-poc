# 유저 문의 1차 분류·답변 자동화 — 설계 문서

> 게임 퍼블리셔 CS 조직의 반복 업무(문의 분류, 답변 작성)를 LLM으로 자동화하는 시스템 설계.
> 목표: 단순 문의의 80%를 AI가 처리하고, CS팀은 어려운 20%에 집중하게 만드는 것.

## 0. 설계 원칙

- **새로 만드는 것 최소화** — 기존 헬프데스크 툴(Zendesk 등)에 LLM을 끼워넣는 구조. CS팀의 워크플로우를 바꾸지 않는다.
- **전부 자동화하지 않는다** — 결제/환불/법적 이슈는 무조건 사람. AI는 초안과 분류로 보조.
- **신뢰를 데이터로 얻고 권한을 늘린다** — 처음엔 초안만 → 정확도 검증 후 단순 카테고리부터 자동 발송.
- **성공 지표는 기술이 아니라 조직 변화** — "분류 정확도 95%"가 아니라 "문의당 처리 시간", "CS 1인당 처리량".

## 1. 기술 스택 / 선정 이유

| 영역 | 선택 | 선정 이유 |
|---|---|---|
| 문의 수집 | Zendesk/Helpshift API + 스토어 리뷰 API | 게임사는 이미 헬프데스크 툴을 운영 중. 새 시스템이 아니라 **기존 워크플로우에 끼어드는** 게 도입 장벽을 낮춤 |
| 분류/답변 생성 | Claude / GPT API (단일 LLM 호출) | 별도 분류 모델 학습 불필요. 프롬프트에 카테고리 정의만 주면 분류 + 답변 초안 + 긴급도 판단을 **한 번의 호출**로 처리. 하이퍼캐주얼 문의는 패턴이 단순(결제, 버그, 광고 보상, 계정)해서 충분 |
| 오케스트레이션 | Python (FastAPI) + 큐 (SQS / Cloud Tasks) | 문의는 실시간성이 필요 없는 비동기 작업. 큐에 쌓고 워커가 처리하면 API 비용·속도 관리가 쉽고 장애 시 재시도 가능 |
| 지식 베이스 | 기존 FAQ/운영 매뉴얼 → **Postgres + pgvector** | 답변 근거를 FAQ에서 검색해 LLM에 주입(RAG) → 환각 방지. 문의량/문서량 규모상 Pinecone 등 전용 벡터 DB는 오버엔지니어링. 운영 DB와 같은 Postgres에서 관리 가능 |
| 모니터링 | 대시보드 (분류 정확도, 자동 답변율, 처리 시간) | 이 시스템의 성공 지표는 "CS팀 처리량 변화" — 측정이 없으면 조직 설득도 없음 |

## 2. 구현 구조

시스템은 두 개의 흐름으로 나뉜다. **(A) 오프라인 인덱싱 파이프라인** — FAQ를 pgvector에 적재하는 배치 흐름. **(B) 온라인 문의 처리 파이프라인** — 문의가 들어올 때마다 도는 실시간 흐름. pgvector는 A의 쓰기 대상이자 B의 읽기 소스로, 두 흐름이 만나는 지점이다.

```
(A) 오프라인: FAQ 인덱싱 파이프라인 (배치 — FAQ 갱신 시마다)
┌─────────────────────────────────────────────────────────┐
│  FAQ / 운영 매뉴얼 / 과거 우수 답변 사례                       │
│        │                                                 │
│        ▼  청킹 (문의유형 단위) → 임베딩 API                   │
│        ▼                                                 │
│  ┌──────────────────────┐                                │
│  │  Postgres + pgvector  │  faq_chunks(content,           │
│  │  (지식 베이스)          │   embedding, game_id, lang)    │
│  └──────────┬───────────┘                                │
└─────────────┼────────────────────────────────────────────┘
              │ (B)-3에서 유사도 검색으로 조회됨
              │
(B) 온라인: 문의 처리 파이프라인 (문의 1건 단위)
              │
[1. 유입]     │
 Zendesk / 인앱 문의 / 스토어 리뷰
        │ (webhook)
        ▼
[2. 수집 레이어]
 문의 정규화: 언어 감지, 메타데이터 첨부 (게임명, 앱 버전, 유저 결제 이력)
        │
        ▼
[큐: SQS / Cloud Tasks] ── 비동기 처리, 장애 시 재시도
        │
        ▼
[3. LLM 처리 워커]
        │
        ├─ (3-1) 문의 본문 임베딩 생성
        │
        ├─ (3-2) pgvector 유사도 검색 ◄──── (A)의 지식 베이스
        │        SELECT ... ORDER BY embedding <=> query_vec LIMIT 5
        │        (game_id, 언어로 사전 필터링 후 top-5 FAQ 청크 조회)
        │
        └─ (3-3) LLM 호출 — 프롬프트 구성:
                 [문의 본문 + 유저 메타데이터 + 검색된 FAQ 청크 top-5]
                 → 한 번의 호출로 3가지 산출:
                   ① 카테고리 (결제/버그/광고보상/계정/기타)
                   ② 긴급도 (환불 요청·결제 오류 = high)
                   ③ 답변 초안 (검색된 FAQ를 근거로 작성,
                      근거 FAQ ID를 함께 출력 → 검증 가능)
        │
        ▼
[4. 라우팅 분기]
        ├─ 신뢰도 높음 + 단순 문의(FAQ성) ──→ 자동 답변 발송
        ├─ 결제/환불/법적 이슈 ──→ 무조건 사람에게 (초안 첨부)
        └─ 신뢰도 낮음 / 검색 결과 빈약 ──→ CS팀 큐로 (분류 태그만 부착)
        │
        ▼
[5. 피드백 루프]
 CS팀이 초안을 수정해 발송 → 수정 전/후 diff 수집
        ├─→ 프롬프트 개선 근거
        └─→ 우수 답변 사례로 (A) 인덱싱 파이프라인에 재투입
             (pgvector 지식 베이스가 운영하면서 점점 좋아지는 구조)
```

### pgvector가 개입하는 지점 정리

| 시점 | 역할 | 동작 |
|---|---|---|
| 배치 (A) | **쓰기** | FAQ·매뉴얼·우수 답변 사례를 청킹 → 임베딩 → `faq_chunks` 테이블에 적재 |
| 문의 처리 (B)-3-2 | **읽기** | 문의 임베딩으로 코사인 유사도 검색(`<=>`), game_id·언어 필터 후 top-5 청크를 LLM 프롬프트에 주입 |
| 라우팅 (B)-4 | **신호** | 유사도 점수가 전부 낮으면 "지식 베이스에 없는 새로운 유형" → 자동 발송 막고 사람에게 라우팅 |
| 피드백 (B)-5 | **성장** | CS팀 수정 답변이 지식 베이스로 재투입 → 검색 품질이 운영과 함께 향상 |

핵심: pgvector는 단순 "FAQ 저장소"가 아니라 **(1) 환각 방지를 위한 답변 근거 공급, (2) 자동 발송 여부를 결정하는 신뢰도 신호, (3) 피드백 루프의 축적 지점** — 세 가지 역할을 동시에 한다.

## 3. 단계적 롤아웃

| 단계 | AI 권한 | 전환 조건 |
|---|---|---|
| Phase 1 | 분류 + 초안만 생성, 발송은 100% 사람 검토 | — |
| Phase 2 | 단순 카테고리(FAQ성)만 자동 발송 | Phase 1에서 해당 카테고리 초안 수정률 < X% |
| Phase 3 | 자동 발송 범위 확대, 다국어 1차 대응 | 카테고리별 수정률 데이터로 판단 |

## 4. 성공 지표 (조직 변화의 언어로)

- 문의당 평균 처리 시간: X분 → Y분
- CS 1인당 일일 처리량: N배
- 자동 답변 비율 / 자동 답변 후 재문의율 (품질 가드)
- 다국어 문의 1차 응답 커버리지 (글로벌 퍼블리셔 셀링 포인트: 언어별 CS 인력 없이 1차 대응)
